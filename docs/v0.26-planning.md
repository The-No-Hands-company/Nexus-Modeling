# v0.26 Planning — Half-Edge Mesh, Robust Predicates, Constraint DOF Analysis

Status: **Planning**. Three geometry kernel correctness upgrades targeting the gap
between scaffolding-grade and production-grade implementations.

Reference: [ROADMAP.md](../ROADMAP.md), [v0.25-planning.md](v0.25-planning.md).

---

## Motivation

The geometry kernel currently has three structural correctness gaps that limit
production use:

1. **Mesh topology** is an indexed polygon soup with no persistent adjacency. Interactive
   modeling operations (edge flip, collapse, split, vertex star iteration) require
   O(n) scans per query. A half-edge structure provides O(1) adjacency for all
   standard mesh editing operations.

2. **Boolean operations** use approximate IEEE 754 arithmetic for all geometric
   predicates. Near-degenerate inputs (nearly coplanar faces, nearly coincident
   vertices) produce incorrect signs that corrupt the output mesh topology.
   Shewchuk adaptive predicates provide exact sign computation with the same
   average-case performance as floating-point.

3. **Constraint solver** runs its iterative relaxation loop blind — it cannot
   detect over-constrained systems (conflicting constraints) or under-constrained
   systems (free DOF remain). Production parametric modelers report constraint
   status to the user as a first-class diagnostic.

## DOF Cost Model

| Constraint Type | DOF consumed |
|---|---|
| `Coincident(A, B)` | 3 — all three coordinates of B fixed to A |
| `Distance(A, B, d)` | 1 — Euclidean distance fixed |
| `AxisAlignedDistance(A, B, axis, d)` | 1 — one coordinate difference fixed |

`remainingDOF = 3 × entityCount − Σ(constraintDOF)`

Over-constrained: `remainingDOF < 0` OR redundant (duplicate) constraints present.
Well-constrained: `remainingDOF == 0` and no redundant constraints.

## Exit Criteria

1. `HalfEdgeMesh::fromMesh` builds correct twin links for closed and open meshes.
2. `flipEdge` and `splitEdge` maintain half-edge invariants.
3. `orient2d` and `orient3d` return the correct sign for near-degenerate inputs that
   naive floating-point arithmetic gets wrong.
4. `ConstraintGraph::analyseDOF()` correctly classifies over/well/under-constrained systems.
5. `ParametricSolverReport` carries DOF fields from `solve()`.
6. All new tests pass; full suite at 100%.
